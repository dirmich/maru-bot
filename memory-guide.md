전체 방향은 좋습니다.
특히 JSON 세션 파일 → SQLite 통합, 단기/장기 기억 분리, FTS5 기반 1차 RAG, 핵심 지침 별도 저장은 marubot 같은 구조에 잘 맞습니다.

다만 그대로 들어가면 몇 가지 문제가 생길 가능성이 큽니다.
결론부터 말씀드리면:

도입 자체는 문제 없지만, “대화 저장소”와 “RAG 기억 시스템”을 완전히 같은 것으로 취급하면 나중에 품질과 유지보수에서 꼬일 수 있습니다.
그래서 저장, 검색, 기억 추출을 약간 분리해서 설계하는 게 좋습니다.

전체 평가

좋은 점부터 보면:

JSON 파일보다 SQLite가 세션 수 증가에 훨씬 강합니다.

최근 대화는 그대로 붙이고, 오래된 대화는 검색해서 넣는 구조는 컨텍스트 절약에 효과적입니다.

FTS5부터 시작하는 것은 아주 현실적입니다.

MEMORY.md를 DB 테이블로 옮기는 것도 자동화 측면에서 맞는 방향입니다.

그런데 주의할 점은 크게 8가지 정도입니다.

1. “메시지 저장”과 “검색용 청크”를 분리하는 게 좋습니다

지금 계획은 대화 메시지 테이블 자체를 FTS5로 바로 검색하는 쪽에 가깝습니다.
처음엔 가능하지만, 나중에 품질 문제가 생길 수 있습니다.

왜냐하면 실제 대화 메시지는:

너무 짧거나

앞뒤 문맥이 없으면 의미가 약하고

한 메시지만 검색되면 오히려 오답을 유도하기 쉽기 때문입니다.

예를 들어 사용자가
“그거 말한 거 다시 정리해줘”
라고 하면, 과거 메시지 한 줄만 찾는 것으로는 부족할 수 있습니다.

그래서 추천은:

분리 구조

messages : 원본 대화 로그 저장

memory_chunks : 검색용으로 재구성한 문맥 청크

facts 또는 directives : 장기 핵심 기억

즉, 저장은 원문 단위, 검색은 청크 단위가 좋습니다.

메시지를 그대로 FTS 대상으로 삼아도 되지만, 최소한 나중엔 청크 테이블로 갈 수 있게 해 두는 게 좋습니다.

2. 최근 N개만 단기 기억으로 두면 부족할 수 있습니다

최근 10~20개 메시지 방식은 단순하고 좋지만, 실제로는 메시지 수보다 토큰 수 기준이 더 안정적입니다.

왜냐하면:

짧은 메시지 20개와 긴 메시지 20개는 완전히 다르고

도구 호출 결과나 긴 설명이 들어오면 최근 10개만으로도 너무 커질 수 있기 때문입니다.

그래서 추천은:

현재 방식보다 더 좋은 기준

최근 N개 메시지

단, 총 토큰 상한도 함께 적용

중요 system/developer/tool 결과는 별도 우선순위 유지

즉:

기본: 최근 10~20개

추가 조건: 총 토큰 예산 초과 시 오래된 것부터 요약/제외

이렇게 하는 게 좋습니다.

3. 키워드 기반 RAG만으로는 “표현이 달라진 질문”에 약합니다

FTS5는 아주 좋지만 한계도 분명합니다.

예를 들어 과거에는
“GPIO 카메라 워커를 subprocess로 분리하자”
라고 했는데, 나중에
“하드웨어 제어를 코어 밖으로 빼자는 얘기 했었지?”
라고 물으면 키워드가 완전히 겹치지 않을 수 있습니다.

그래서 1단계는 FTS5로 충분하지만, 구조상 나중에 아래가 가능해야 합니다.

이후 확장 경로

FTS5 lexical 검색

태그/메타데이터 필터

선택적 임베딩 검색

hybrid reranking

즉, 지금 당장은 FTS5만 넣되 스키마를 벡터 확장 가능하게 설계하는 건 아주 좋습니다.

4. “관련 메시지 3~5개 주입”은 위험할 수 있습니다

이 부분은 꽤 중요합니다.
그냥 상위 3~5개 메시지를 넣으면 다음 문제가 생깁니다.

서로 다른 시점의 문장이 섞임

원래 맥락과 반대로 해석될 수 있음

outdated 정보가 같이 들어올 수 있음

tool 결과와 사용자 발화가 섞이면 혼란스러움

그래서 추천은 메시지 단위가 아니라 검색 결과를 문맥 블록으로 재구성하는 것입니다.

예를 들면:

hit가 나온 메시지 1개만 넣지 말고

그 주변 2~4개 메시지까지 같이 묶어서

“과거 대화 참고” 블록으로 넣기

즉:

나쁜 방식

메시지 5줄 그냥 주입

좋은 방식

관련도 높은 구간 2~3개를 찾아

각 구간을 짧게 묶어서 넣기

예:

2026-03-01 “RAG 구조 논의”

2026-03-05 “subprocess worker 설계”

2026-03-08 “skill manifest 규칙”

이런 식으로 블록화하는 게 훨씬 좋습니다.

5. MEMORY.md의 자동 저장은 잘못하면 오염됩니다

이 방향은 좋지만 제일 위험한 부분이기도 합니다.

봇이 “중요하다”고 판단해서 자동 저장하면 실제로는:

일시적 취향

맥락 의존 발언

잘못 이해한 내용

이미 바뀐 결정

까지 저장할 수 있습니다.

그래서 핵심 지침 테이블은 그냥 자동 누적하면 안 되고, 최소한 다음 속성이 필요합니다.

필수 필드

kind : preference / rule / project_fact / identity / temporary

confidence

source_message_id

created_at

updated_at

expires_at 또는 stale_after

status : active / superseded / archived

즉, 기억은 “쌓는 것”보다 갱신과 폐기가 중요합니다.

예를 들어:

“앞으로 일본어로 답해줘”

“이번 대화에서는 한국어로 해줘”

이 둘은 충돌할 수 있습니다.
그래서 무조건 저장보다 우선순위와 유효기간 개념이 있어야 합니다.

6. SQLite 하나에 모든 걸 몰아넣어도 되지만 쓰기 경쟁은 조심해야 합니다

marubot이 단일 프로세스 중심이면 SQLite가 잘 맞습니다.
하지만 나중에 다음이 동시에 붙으면 주의가 필요합니다.

세션 저장

FTS 인덱스 갱신

memory fact 업데이트

background summarization

dashboard 조회

worker 로그 기록

이게 한 DB에 몰리면 write contention이 생길 수 있습니다.

그래서 추천은:

최소 권장

WAL 모드 사용

짧은 트랜잭션 유지

배치 삽입 사용

인덱스 갱신 시점 제어

더 나은 구조

대화 저장은 즉시

무거운 memory extraction / chunk generation은 비동기 작업 큐로 처리

즉, 사용자 메시지 처리 경로에서 모든 후처리를 동기적으로 하지 않는 게 좋습니다.

7. “1주일 전 대화를 기억”은 검색만으로 안 될 수 있습니다

검증 계획 자체는 좋지만, 단순히 검색 결과가 나온다고 해서 “정확히 기억한다”고 보긴 어렵습니다.

왜냐하면 LLM이 검색된 내용을:

잘못 요약하거나

현재 질문과 섞거나

오래된 내용을 최신 사실처럼 말할 수 있기 때문입니다.

그래서 검증은 두 층으로 하셔야 합니다.

A. retrieval 평가

정말 관련 메시지를 잘 찾는가

B. answer 평가

찾은 내용을 기반으로 올바르게 답하는가

즉, “찾기”와 “말하기”를 따로 검증해야 합니다.

8. 세션 단위 저장만으로는 부족할 수 있습니다

현재 설명상 세션별 대화 저장소에서 전체 과거 대화를 찾는 구조인데, 나중에는 사용자가 세션을 넘어서 같은 프로젝트를 이어갈 수 있습니다.

예를 들어:

어제는 marubot worker 설계

오늘은 manifest schema

내일은 RAG 개선

이게 서로 다른 세션이라도 같은 주제일 수 있습니다.

그래서 스키마에는 최소한 이런 메타데이터가 있으면 좋습니다.

추천 메타데이터

session_id

conversation_id 또는 topic_id

role

timestamp

project

source_type : user / assistant / tool / system

이렇게 해 두면 나중에 “현재 프로젝트와 관련된 과거 문맥만” 검색하기 쉬워집니다.

추천하는 더 안전한 구조
1단계: 저장소 교체

좋습니다. 다만 테이블을 이렇게 나누는 걸 권합니다.

sessions

retrieval_cache 또는 memory_chunks 준비용 테이블

2단계: FTS5 기반 retrieval

좋습니다. 단, 검색 대상은 가능하면 나중에 messages에서 memory_chunks로 확장 가능하게 하세요.

검색 결과는:

상위 메시지 3~5개 그대로가 아니라

상위 관련 구간 2~3개로 재구성

이게 좋습니다.

3단계: 핵심 기억 자동화

좋습니다. 다만 “자동 저장”보다
후보 추출 → 점수화 → 저장 → 나중에 갱신/폐기 구조가 안전합니다.

제가 권하는 실제 우선순위
꼭 먼저 할 것

JSON → SQLite migration

retrieval 결과를 문맥 블록으로 재조합

바로 넣지 않는 게 좋은 것
