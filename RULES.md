# MaruBot 프로젝트 개발 및 릴리즈 규칙 (Development & Release Rules)

이 문서는 MaruBot 프로젝트에 참여하는 모든 AI 에이전트와 개발자가 준수해야 할 핵심 규칙을 정의합니다. 새로운 세션 시작 시 항상 이 규칙을 확인하고 적용해야 합니다.

## 1. 표준 배포 워크플로우 (Mandatory Release Workflow)

모든 기능 작업이 완료된 후, 아래 **6단계 절차**를 반드시 순서대로 수행해야 합니다:

1. **전방위 버전업**: `versionup.md` 문서를 **반드시 먼저 읽고** 그 절차에 따라 모든 파일의 버전을 동일하게 올립니다.
   - 대상: `pkg/config/version.go`, `README.md` (및 모든 언어별 README), `IDENTITY.md`, `Project.md`, `web-admin/package.json`
2. **내부 저장소 동기화**: `maruminibot` 저장소에 **한글** 커밋 메시지로 `git add/commit/push`를 수행합니다.
3. **웹 관리자 빌드**: `maruminibot`의 `web-admin` 폴더에서 `npm run build`를 수행하고 `make sync-ui`로 백엔드에 반영합니다. (매우 중요: 공개 저장소에는 소스가 없으므로 여기서 반드시 빌드를 완료해야 함)
4. **바이너리 빌드**: `go build`를 통해 최신 버전의 Windows 실행 파일(`.exe`)을 새로 생성합니다.
5. **공개 저장소 동기화**: `../maru-bot` 저장소로 모든 최신 소스와 빌드 결과물(특히 `web-admin/dist`)을 복사(Publish)합니다.
6. **공개 저장소 최종 배포**: `../maru-bot` 저장소에 **영어** 커밋 메시지로 `git add/commit/push`를 수행하고 GitHub Release를 생성합니다.

## 2. 언어 및 커밋 메시지 규칙
- **내부 커뮤니케이션**: 모든 문서(`task.md`, `implementation_plan.md`, `walkthrough.md`)와 사용자 알림(Notification)은 **한글**을 사용합니다.
- **커밋 메시지**:
  - `maruminibot` (Private): **한글** 메시지 사용. (윈도우 환경에서는 `chcp 65001` 또는 메시지 파일 사용 권장)
  - `maru-bot` (Public): **영어** 메시지 사용.

## 3. 설정 및 환경 관리
- **하위 호환성**: 설정 구조 변경 시 기존 `config.json` 형식을 새로운 구조로 자동 전환하는 마이그레이션 로직을 반드시 포함합니다.
- **민감 정보**: `publish.sh`를 통해 공개 저장소로 배포할 때 개인 설정 파일(`usersetting.json` 등)이 포함되지 않도록 주의합니다.

## 4. 도구 및 빌드
- **UI 빌드**: 모든 배포 전 반드시 `web-admin` 폴더에서 `npm run build`를 수행하여 최신 UI가 포함되도록 합니다.
- **바이너리 배포**: 공개 저장소 및 GitHub Release용 바이너리는 **Windows**와 **macOS**용만 생성합니다. 
  - **Windows**: `.zip` 파일은 생성하지 않으며, 실행 파일(`.exe`)만 배포합니다.
  - **macOS**: 개별 실행 파일(binary)은 배포하지 않으며, 최종 설치 패키지인 `.dmg` 파일만 배포합니다.
  - Linux 사용자는 `install.sh` 또는 `maru-setup.sh`를 통해 소스 빌드 방식으로 설치하므로 별도 바이너리를 배포하지 않습니다.
- **검증**: 커밋 전 반드시 `go build`를 통해 빌드 오류가 없는지 확인합니다.
- **Windows 인코딩**: 한글 커밋 메시지가 깨지지 않도록 항상 `chcp 65001`을 먼저 실행하거나, `-F` 옵션을 통해 UTF-8 인코딩된 파일로부터 메시지를 읽어옵니다.

## 5. 자가 검증 및 완결성 (Self-Verification & Completion)
- **완결성**: 모든 배포 작업 시 `maruminibot`과 `maru-bot` 두 저장소 모두에 대해 다음 사항을 **말하지 않아도** 완벽하게 처리합니다:
  - 소스 코드 커밋 및 푸시
  - 버전 태그 생성 및 푸시
  - `make public`을 통한 공개 자산(Asset) 릴리즈 확인
- **자가 검증**: 보고 전 인접 저장소의 `git status`와 `git tag`를 직접 조회하여 누락된 단계가 없는지 스스로 확인합니다. "끝났다"고 보고하는 시점에 모든 원격 저장소와 배포판이 최신 상태여야 합니다.

## 6. 버저닝 및 글로벌 문서 관리 (Versioning & Global Docs)
- **README 동기화**: 버전 업데이트 시 `README.md` (KO), `README-en.md`, `README-ja.md`, `README-cn.md` 등 모든 글로벌 문서의 버전 번호와 주요 변경 사항을 동일하게 업데이트합니다.
- **홈페이지 연동**: 새로운 릴리스가 있을 때마다 `homepage`에서도 최신 버전 정보를 반영하여 사용자가 항상 최신 버전을 안내받을 수 있도록 합니다.
- **기록 의무**: `Project.md`에 모든 버전 이력을 남기는 것은 필수이며, 이는 자동화된 문서 생성의 기초가 됩니다.

## 7. 공개 배포 및 리포지토리 관리 (Public Release & Repo Management)
- **배포 대상**: 공식적인 모든 공개용 바이너리와 소스 코드는 반드시 **`maru-bot` 리포지토리**를 통해 배포합니다. 
- **프라이빗 리포지토리 제한**: 내부용인 **`maruminibot` 리포지토리에는 절대 태그(Tag)를 생성하거나 릴리즈(Release)를 만들지 않으며, 바이너리 또한 업로드하지 않습니다.** 모든 버전 관리와 배포 자산은 오직 `maru-bot`에서만 관리합니다.
- **기본 언어**: 공개 배포 시 Web Admin의 기본 언어는 **영어(en)**로 설정되어야 합니다.
- **버전 표기 (Versioning)**: 기존 관례를 따라 버전 태그 및 릴리즈 명칭에 **`v` 접두사를 붙이지 않습니다.** (예: `v0.4.86` (X) -> `0.4.86` (O))
- **릴리즈 클린업**: 새로운 릴리즈 생성 시 중복된 태그가 생기지 않도록 주의하며, 불필요하거나 잘못된 형식의 태그는 즉시 삭제합니다.
- **포트 설명**: Webhook 등 네트워크 관련 설정 시 `0` 입력이 기본 게이트웨이 포트 사용을 의미한다는 안내를 UI와 문서에 포함합니다.
